# Code Review: Ilex Inventory Frontend

## Persona

Pragmatic reviewer. Knows this project's architecture cold and won't accept drift. Architectural violations and locked-decision relitigations are non-negotiable; style is suggested. Explain *why*, not just *what*. When the user (or an automated downstream step) asks "what should I auto-fix?", mark each finding `auto-fixable: yes/no` so the fixer doesn't have to guess.

## Stack Context

React 18, Vite 5, TypeScript 5.5 strict, TanStack Router (file-based, generated `src/routeTree.gen.ts`), TanStack Query 5, Mantine 7 + Tailwind 3, Decimal.js, openapi-fetch + openapi-typescript (codegen from the BE's drf-spectacular schema), Zustand for client state, msw + Vitest for tests, Playwright for E2E smoke. Cookie-based session auth (BE's `sessionid` + `csrftoken`); the FE is a SPA at a different origin from the BE in production (Netlify ↔ Railway).

The complete rule set is in `.claude/skills/ilex-discipline/SKILL.md` and the test discipline in `.claude/skills/tdd/SKILL.md`. This skill applies them as review criteria.

## Review Process

### 1. Layer discipline (highest priority)

Data flows **Routes → Features → Data → Generated client**. No imports upward.

| Layer | Owns | Forbidden |
|---|---|---|
| `src/routes/*.tsx` | Path declaration + page mount + optional TanStack Query loader. One file per page. | Inline forms, business logic, direct API client calls, direct `src/api/generated/` imports. |
| `src/features/{domain}/` | Page components, domain-specific UI (FEFO preview, recall toggle), forms (validate locally, submit via mutation hooks). May import from `src/data/{domain}/` and `src/components/`. | Direct `src/api/generated/` imports. Bare `fetch` / `axios`. Raw CSS modules outside `src/theme/`. |
| `src/data/{domain}/` | `queries.ts`, `mutations.ts`, `keys.ts`. Wraps TanStack Query. Mints `Idempotency-Key` (UUIDv7) on terminal endpoints. Maps 4xx envelope to `ApiError`. May import from `src/api/`. | Rendering UI. Returning JSX. |
| `src/api/client.ts` | Thin openapi-fetch wrapper. `credentials: 'include'`, CSRF (X-CSRFToken from in-memory store / cookie), Idempotency-Key plumbing, 4xx normalization. | Domain-specific paths or types — those live in `src/data/`. |
| `src/api/generated/` | Regenerated by `npm run generate:api`. Committed. | Hand-edits. Parallel-defined request/response shapes anywhere else. |

**What to flag — Critical / auto-fixable: YES:**
- `import … from '@/api/generated/…'` inside `src/features/` or `src/routes/` → route via a `src/data/{domain}/` hook
- `fetch(` or `axios.` outside `src/data/`, `src/api/`, `src/utils/csv-export.ts` → move to a data hook
- Page component (`*Page.tsx`) calling `apiClient.GET/POST/...` directly → wrap in a `use{Noun}` hook in `src/data/`
- Route file (`src/routes/*.tsx`) containing form JSX or `useMutation` calls → move to a feature page

**What to flag — Critical / auto-fixable: NO:**
- Hand-written types parallel to a generated schema (likely needs BE schema fix + regen)
- New cross-feature import (e.g. `src/features/sales/` reaching into `src/features/inventory/foo`) → likely needs a shared hook in `src/data/` or a primitive in `src/components/`

### 2. Generated client is never hand-edited

`src/api/generated/schema.ts` and `src/api/generated/index.ts` are regenerated by `npm run generate:api`. CI gate: `npm run generate:api -- --check` returns no diff.

**Critical / auto-fixable: YES:**
- Modification anywhere in `src/api/generated/` → run `npm run generate:api` against the live BE; if drift remains, update `docs/openapi.snapshot.json` + regen
- Hand-written request/response type that duplicates a generated one (e.g. `type ProductCreateRequest = { ... }` outside generated) → import the generated type instead

**Critical / auto-fixable: NO:**
- Generated client *and* hand-typed shape both used in the same data layer (`src/data/auth/types.ts` is the documented sole exception — BE auth endpoints lack OpenAPI bodies). Adding a second exception requires explicit sign-off.

### 3. Money / quantity as strings end-to-end

`numeric(14, 4)` on the BE → string in TypeScript. `Decimal.js` for arithmetic. JS `number` math on a money or qty value is a critical bug.

**Critical / auto-fixable: YES:**
- `parseFloat(x)`, `Number(x)`, or `+x` where `x` is a `quantity`, `unit_cost`, `unit_price`, `sell_price`, `signed_quantity`, `total`, `revenue`, `cogs`, `profit`, `on_hand`, `allocated_quantity`, etc. → wrap in `new Decimal(x)`; format via `formatMoney` / `formatQty`
- Multiplication / addition with `*` or `+` between two money/qty strings → use `Decimal.js`: `new Decimal(a).times(b).toFixed(4)`
- A money/qty form field bound to `<NumberInput>` (which returns `number`) → `<DecimalInput>` (string round-trip)

**Critical / auto-fixable: NO (review needed):**
- `<NumberInput>` in `src/features/` for a non-money/qty integer field (page size, `expiring_within` days). Allowed by gate-5 allowlist but adding a new allowlisted file requires a `scripts/check-grep-gates.sh` update.

### 4. Terminal mutations never optimistic

The BE walks FEFO + writes allocations + ledger movements + idempotency rows in one transaction. The client cannot honestly mirror that. **Receive PO, commit SO, void SO, recall, un-recall, write-off, manual batch, products import** all use no-optimistic mutations: invalidate + refetch on success.

**Critical / auto-fixable: YES:**
- `onMutate: async () => queryClient.setQueryData(...)` on any of the eight terminal mutations → drop `onMutate`; rely on `onSuccess` to invalidate
- A terminal mutation hook missing `onSuccess: () => invalidateQueries({...})` → add the invalidation

**Warning / auto-fixable: YES:**
- Optimistic update on a non-terminal PATCH that touches more than one row's display fields → safer to invalidate

### 5. Idempotency-Key on terminal POST

`src/api/client.ts` has `ALWAYS_IDEMPOTENT_POST_PATHS` (the seven BE-required endpoints) + `CONDITIONAL_IDEMPOTENT_POST_PATHS` (`/batches/{batch_id}/movements` when `kind === 'write_off'`). Caller-supplied key always wins so retry semantics work.

**Critical / auto-fixable: YES:**
- New BE-required terminal endpoint shipped on the FE without being added to `ALWAYS_IDEMPOTENT_POST_PATHS` → add the schema path
- Hardcoded `Idempotency-Key` value (other than test fixtures) — a UUIDv7 must be minted per attempt

**Mandatory test (gate via `tdd` skill):**
- Every terminal mutation hook has a test asserting the `Idempotency-Key` header is present, non-empty, and stable across automatic retries within one user-initiated attempt

### 6. Cross-owner is 404 (BE-D4)

The BE returns 404 for cross-owner reads and writes (never 403). The FE must surface "Not found" without distinguishing from never-existed. `useQuery`'s `retry` should not retry on 404.

**Critical / auto-fixable: YES:**
- Detail-page `useQuery` without `retry: (failureCount, error) => !(ApiError.is(error) && error.status === 404) && failureCount < 3` — add it
- Distinguishing copy ("you don't have permission" vs "not found") on a 404 page → unify to "Not found"

**Mandatory test:**
- Every detail route has a 404 variant test (server returns 404 → page renders "Not found")

### 7. CSRF + cookie auth

Cross-origin (Netlify ↔ Railway): JS at the FE origin **cannot read** the `csrftoken` cookie set by the BE via `document.cookie` — fundamental browser security boundary. The token rides through the response body of `/auth/me`, `/auth/login`, `/auth/signup`; FE stashes it in `src/api/csrf-store.ts` and the apiClient middleware echoes it back as `X-CSRFToken` on POST/PATCH/DELETE.

**Critical / auto-fixable: YES:**
- New auth-adjacent endpoint that returns user state but omits stashing `csrf_token` from its response → wire through `setCsrfToken()`
- apiClient middleware reading only from `document.cookie` (not also `getCsrfToken()`) → fall through to cookie only as a same-origin fallback

**Critical / auto-fixable: NO:**
- Adding new `credentials: 'include'` exemptions (every request needs the session cookie)

### 8. 4xx envelope normalization

Every BE 4xx returns `{ error: ErrorCode, detail?: string, fields?: Record<string, string> }`. `apiClient`'s response middleware maps that into a typed `ApiError`. Features render:
- `fields` → field-level form messages
- `detail` → toast on non-form errors
- 409 on terminal mutations → refetch the affected detail + toast `"This {resource} has already been {received|committed|voided|...} elsewhere."`
- 404 on detail routes → "Not found"

**Critical / auto-fixable: YES:**
- Catching a raw `Error` with no `ApiError.is()` guard in a feature → adopt the typed branch
- `mutation.error?.message` (string) instead of `(mutation.error as ApiError).fields?.foo` → use the typed accessor
- Generic `"Something went wrong"` toast where the envelope already has a `detail` → render `detail`

### 9. Imports — module-top only

Function-local `import` statements are forbidden in non-test code (matches BE invariant). Tests are not gated but should also keep imports module-top. Type-only imports must use `import type` where possible to keep the runtime bundle clean.

**Critical / auto-fixable: YES:**
- Indented `import` inside a function or callback in `src/` → hoist to module top, dedupe, sort (external → `@/` → relative)
- Default `import` of a type → switch to `import type`

### 10. Behavioral tests only (TDD skill)

Outcomes only: rendered DOM, observable state, MSW request shape, `Idempotency-Key` on the request, navigation result. Never test internals.

**Critical / auto-fixable: YES (test deletion requires confirmation):**
- `vi.mock('@/data/...')` mocking a hook that the component under test owns → use MSW to drive the network, real QueryClient
- Asserting on `result.current.queryKey` or other internal Query state → assert the rendered DOM or the next request
- Snapshot tests for component output → replace with role/text-based assertions

**Critical / auto-fixable: NO:**
- Echo tests across layers (same behavior asserted at hook + page) — pruning requires judgment about which layer owns the case

### 11. MSW handlers in tests

Tests describe state declaratively via MSW handlers (`http.get(...)`, `http.post(...)`). Handlers must:
- Use absolute URLs (`http://localhost:8000/api/v1/...`) — `vitest.config.ts` forces `VITE_API_BASE_URL` to that origin so openapi-fetch + MSW agree
- Return the BE's exact response shape (matched against `docs/openapi.snapshot.json`)
- Match Idempotency-Key / X-CSRFToken assertions where the test verifies them

**Critical / auto-fixable: YES:**
- Handler returning a hand-fabricated shape that doesn't match the BE's response (e.g. nested `lines[].allocations` after the BE moved allocations to top-level) → reshape to match the snapshot
- Relative URL in a handler (`http.get('/api/v1/...')`) → prefix with `http://localhost:8000`

### 12. Function size (max 60 lines)

Strict bar (matches BE). A component or hook over 60 LOC is split or extracted. Common extractions in this codebase:
- Modal-and-form pairs split into `<XConfirmModal>` and `<XForm>` siblings
- TanStack Query options into a typed `*QueryOptions` object
- URL builders (`buildSosListUrl(params)`) for routes with cursor + filters

**Critical / auto-fixable: YES:**
- Function or component ≥ 60 lines → extract sibling components or named handlers within the same file. Tests must stay green.

### 13. SOLID — SRP only

Functional components with hooks. SRP is the load-bearing principle. OCP/LSP/ISP/DIP are not load-bearing — don't introduce factory hooks, render-prop strategies, or DI containers to "satisfy" them.

**Critical / auto-fixable: YES:**
- A component clearly doing 5+ steps that map to extractable sibling components (header + table + footer + modal + side-effect) — extract.

### 14. DRY — known duplication targets

Recurring near-duplicates already known:
- Status badge color mappers (`statusBadgeColor`) duplicated in `SosListPage.tsx` + `SoDetailPage.tsx` → consider one helper in `src/features/sales/utils.ts`
- "Loading skeleton + error alert + empty state" wrapper across list pages — consolidated to `src/components/{LoadingSkeleton,EmptyState,ErrorBoundary}.tsx`. New list pages should reuse, not re-implement.
- `effectiveStatus(so)` for voided-at derivation duplicated in `SosListPage` + `SoDetailPage` → one helper

**Warning / auto-fixable: NO:**
- Don't create cross-feature abstractions chasing DRY. Same-feature duplication can be extracted; cross-feature similarity is usually coincidence (different data shapes).

### 15. Domain correctness

- **FEFO ordering**: preview allocations must render `expiration_date ASC NULLS LAST`. The BE provides them in that order; client sorts must agree. `<14d` to expiration → `<ExpiryBadge>` (orange) on the row.
- **Voided SOs keep `status='committed'`**: the void signal is `voided_at != null`. List + detail pages must derive display via `effectiveStatus(so)`; reading `status` alone shows the wrong badge.
- **Allocations are top-level on `SalesOrderResponse`** (not nested per-line, post-BE drift). Per-line read crashes with "Cannot read properties of undefined (reading 'length')".
- **Recall is a terminal action with required reason** (BE-D3). Unguarded `Switch`/`Toggle` is wrong; use a confirm modal.
- **Margin formula is markup**: `(revenue - cogs) / cogs × 100%` (BE-D13). $1,000 / $100 → 900%, not 90%.
- **Light mode does not exist**. Charcoal-only palette. No `:root` light variants.
- **Fonts are Inter (UI) + JetBrains Mono (data)**. Geist appears in v0 prototype only.

### 16. Naming

| Item | Convention |
|---|---|
| Route file | `_authed.{domain}.{action}.tsx` (e.g. `_authed.purchase-orders.new.tsx`) |
| Page component | `{Domain}{Action}Page.tsx` (e.g. `SosListPage`, `SoDraftPage`, `SoDetailPage`) |
| Feature component | `{Noun}{Role}.tsx` (e.g. `FefoPreview`, `CommitConfirmModal`) |
| Query hook | `use{Noun}` / `use{Noun}List` (e.g. `useSo`, `useSosList`) |
| Mutation hook | `use{Verb}{Noun}` (e.g. `useCommitSo`, `useReceivePo`) |
| Cache key namespace | `{domain}Keys.{noun}({...args})` (e.g. `salesKeys.detail(id)`) |
| Test file | `{Subject}.test.tsx` (colocated; pages live under `__tests__/`) |
| Generated module | `src/api/generated/schema.ts` (single emit) |

### 17. Errors — `ApiError`

Defined in `src/api/errors.ts`. Constructed with `{ status, error, detail?, fields? }`. Type guard: `ApiError.is(e)`. Features should never inspect raw `Error.message`.

### 18. Tailwind + Mantine

- Mantine for layout primitives (Stack, Group, Card, Modal, Spotlight, ...) and form inputs (`TextInput`, `Select`, `DateInput`, `Checkbox`).
- Tailwind for spacing tweaks, one-offs, and the `floor:` mode variant (`addVariant('floor', '&.floor, .floor &')`).
- Theme tokens live in `src/theme/tokens.ts` and feed both Tailwind (`tailwind.config.ts`) and Mantine (`src/theme/mantine.ts`).
- Charcoal-only — no light theme.

**Critical / auto-fixable: YES:**
- Inline `style={{ color: '#XYZ' }}` for a brand color → import the token from `tokens.ts` or use a Tailwind class
- Raw CSS module file outside `src/theme/` → move to Tailwind / token-driven Mantine prop
- A new color hex literal anywhere in `src/features/` → hoist into `tokens.ts`

### 19. Floor mode

A `<html class="floor">` toggle. Persisted in localStorage. Pure CSS via the Tailwind `floor:` variant. The API has no awareness of mode.

**Critical / auto-fixable: NO:**
- API request altering behavior based on floor mode → architectural drift; lift to a feature flag UI without sending state to the BE.

### 20. Commits

- **Conventional Commits** title only. Format: `<type>(<scope>): <short description>` (e.g. `feat(sales): commit endpoint with FEFO walk + idempotency`).
- Body acceptable for non-trivial changes (FE convention diverges from BE here — the FE keeps `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>` per project history; BE strips it).
- One commit per issue when possible; split only when scopes are genuinely independent.

## Severity & auto-fix policy

| Severity | Definition | Default auto-fix |
|---|---|---|
| Critical, auto-fixable: YES | Mechanical drift from a hard rule (grep gate, layer rule, money/qty discipline, idempotency plumbing, generated client edits) | Apply the fix, run `npm test && npm run typecheck && npm run lint`, commit |
| Critical, auto-fixable: NO | Architectural redesign or test semantics that need judgment (cross-feature reach, schema drift requiring BE coordination) | Report, halt, ask user |
| Warning, auto-fixable: YES | Quality / DRY / function-size | Apply if scope is small (one file, < 30 LOC of change) |
| Warning, auto-fixable: NO | Judgment call (echo tests, naming nits) | Report only |
| Note | Style or future-proofing | Report only |

## Important

- **Diff scope**: review only the changes the user asks about. Default: `git diff <last-completed-commit>..HEAD --stat` then walk each file. For a per-issue review, the issue's predecessor commit is the diff base — see `docs/issues/status.md` for the issue-to-completion log.
- **Whole-repo mode**: when the user explicitly asks for a full sweep ("review the entire codebase"), walk every file under `src/` and `tests/` against this skill's rules, then report against the same format with the metrics block sourced from real commands.
- **Don't relitigate locked decisions**. SPEC §2.5 / BE-D# / FE invariants 1–8 are settled; flag the violation, don't re-debate the rule.
- **Verify before claiming clean**. Run `npm test`, `npm run typecheck`, `npm run lint`, `npm run generate:api -- --check`, and the discipline greps from `ilex-discipline`'s "CI gates" section. The report's metrics block must come from real commands, not reasoning.
- **Commit-ready output**. After fixes are applied and the validation chain is green, the report ends with the suggested Conventional Commit message and a `🟢 ready to commit` marker. If anything is red, end with `🔴 do not commit — <reason>`.

## Quick command reference

```bash
# Validation chain (all must pass)
npm run typecheck
npm test
npm run lint
npm run generate:api -- --check
npm run build

# Discipline greps (lint:gates already bundles 1–6; included here for ad-hoc use)
grep -RE "(fetch|axios)\(" src/ --include='*.ts' --include='*.tsx' \
  | grep -vE "src/data/|src/api/|src/utils/csv-export\.ts"
grep -RE "as any" src/ --include='*.ts' --include='*.tsx' \
  | grep -v "src/api/generated"
grep -RE "from ['\"].*api/generated" src/features/ src/routes/
grep -RE "\bNumberInput\b" src/features/
grep -RE "console\.(log|debug)" src/ --include='*.ts' --include='*.tsx'

# Function-size scan (rough; reports files with > 60-line functions)
find src -name '*.ts' -o -name '*.tsx' \
  | xargs awk '/^(function|const .* =|export function|export const)/{name=$0; start=NR} \
               /^}$/{if (NR - start > 60) print FILENAME":"start" "name" ("NR-start" lines)"}'

# Generated client + snapshot drift (requires live BE URL or local snapshot)
VITE_OPENAPI_URL="https://<be-host>/api/v1/openapi.json?format=openapi-json" \
  npm run generate:api -- --check
```
